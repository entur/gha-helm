## Project overview

`entur/gha-helm` provides reusable GitHub Actions workflows for Helm operations (lint, deploy, unit test). It is consumed by many teams across Entur via `uses: entur/gha-helm/.github/workflows/<workflow>.yml@v1`.

## Repository structure

```
.github/
  actions/
    helm-lint/action.yml      # Composite action: helm dependency update + lint
    helm-deploy/action.yml    # Composite action: helm upgrade --install with rollback
  workflows/
    lint.yml                  # Reusable workflow: Helm lint (user-facing)
    deploy.yml                # Reusable workflow: Helm deploy (user-facing)
    unittest.yml              # Reusable workflow: Helm unit test (user-facing)
    ci.yml                    # CI pipeline (internal, runs on PRs)
    cd.yml                    # CD/release pipeline (internal, runs on main)
fixture/helm/                 # Test charts used by CI
  amazing-app/                # Standard GCP chart
  amazing-app-az/             # Azure chart with unit tests
  amazing-app-multi/          # Multi-app deployment chart
  amazing-app-containers/     # Containers array chart (name-based image replacement)
README-lint.md                # Lint workflow docs (auto-generated sections)
README-deploy.md              # Deploy workflow docs (auto-generated sections)
README-unittest.md            # Unit test workflow docs (auto-generated sections)
```

## Key concepts

- **User-facing workflows**: `lint.yml`, `deploy.yml`, `unittest.yml` -- these are the public API consumed by other repos.
- **Composite actions**: `helm-lint` and `helm-deploy` contain the actual logic, invoked by the workflows above.
- **Cloud providers**: Supports GCP (default) and Azure (`cloud_provider: az`). Auth and registry differ per provider.
- **Rollback**: On deploy failure, `helm-deploy` automatically rolls back and reports status codes (0=success, 1=failed, 2=rolled back, 3=rollback failed).
- **Multi-app deploy**: Supports comma-separated `image_set_path` for charts with multiple aliased dependencies.
- **Container name deploy**: `container_name` resolves a container by name in `common.containers` array, avoiding fragile array indices.
- **Slack notifications**: Optional via `slack_channel_id` input, includes approval gate detection and deploy result messages.
- **Image skip**: Setting `image: na` skips image replacement in Helm values.
- **Analytics**: Workflows send usage data to PostHog via `entur/gha-meta/.github/actions/posthog`.

## Development workflow

### Conventions

- Use [conventional commits](https://www.conventionalcommits.org/): `feat:`, `fix:`, `chore:`, `docs:`, `style:`, `refactor:`.
- `feat:` triggers a minor version bump, `fix:` triggers a patch bump.
- Dependabot PRs for user-facing workflows use `fix(deps):` to trigger releases.
- Pin third-party actions to commit SHAs with version comments (e.g. `@sha # v5.0.0`). Do **not** pin `entur/` actions to SHAs.
- Code is owned by `@entur/team-plattform`.

### Testing

There are no local tests. All testing happens in CI (`ci.yml`) which:
1. Lints fixture charts (GCP, Azure, multi-app, containers)
2. Runs `helm unittest` on fixture charts
3. Deploys fixture charts to real dev clusters (GCP and Azure)
4. Tests rollback by deploying a bad image
5. Tests `image: na` (skip image update)
6. Tests `container_name` (name-based image replacement)
7. Cleans up test releases after each test

To test changes: open a PR and let CI run. The CI deploys to actual Kubernetes clusters.

### Releases

Releases are automated via `cd.yml` using `entur/gha-meta/.github/workflows/release.yml@v1`. Pushing to `main` triggers a release based on conventional commit prefixes.

### Documentation

README files contain `AUTO-DOC-INPUT` / `AUTO-DOC-OUTPUT` sections that are regenerated by CI. Do not edit these sections manually.

## Making changes

- Changes to user-facing workflows (`lint.yml`, `deploy.yml`, `unittest.yml`) affect all consuming repos. Be careful with breaking changes.
- Changes to composite actions (`helm-lint`, `helm-deploy`) are where the core logic lives.
- The `deploy.yml` workflow orchestrates: input validation, cloud auth, k8s auth, then calls the `helm-deploy` composite action.
- Fixture charts in `fixture/helm/` are specifically for CI testing. Match their structure to test new features.
- When adding inputs to workflows, also update the composite action if needed, and let CI regenerate the README docs.
